# План разработки приложения Presentation Maker

Приложение для создания презентаций в формате pptx из текстового описания слайдов и образца слайда в формате pptx.

**Стек:** фронтенд — React, бекенд — FastAPI.

**Контекст:** пользователь задаёт текстовое описание слайдов (заголовок и пункты для каждого слайда) и загружает образец слайда в формате pptx; приложение генерирует готовую презентацию pptx на основе описания и стиля/структуры образца.

---

## Этап 1. Требования и спецификация

- **1.1** Сбор и согласование требований: входные данные (текстовое описание слайдов, загрузка образца pptx), выход (готовый pptx), ограничения, граничные случаи → **business-analyst**
- **1.2** Формулировка user stories и критериев приёмки; приоритизация фич (MVP и последующие итерации) → **business-analyst**
- **1.3** Подготовка ТЗ/PRD и передача в архитектуру; проверка полноты и непротиворечивости требований → **business-analyst**

**Результат:** документ требований (ТЗ/PRD), user stories, критерии приёмки.

---

## Этап 2. Архитектура и технический дизайн

- **2.1** Выбор структуры проекта (монорепо / раздельные репозитории), границы фронтенд/бекенд, API-контракт → **architect**
- **2.2** Проектирование бекенда: эндпоинты (загрузка образца, приём описания слайдов, генерация pptx), работа с pptx (библиотеки, извлечение стиля из образца) → **architect**
- **2.3** Нефункциональные требования: лимиты размера файлов, таймауты, масштабируемость; рекомендации по безопасности (загрузка файлов, валидация) → **architect**
- **2.4** Фиксация решений в ADR (Architecture Decision Records); передача спецификации разработчику и дизайнеру → **architect**

**Результат:** архитектурный документ, ADR, описание API.

---

## Этап 3. UI/UX и дизайн интерфейса

- **3.1** User flows: ввод описания слайдов, загрузка образца, запуск генерации, скачивание результата, обработка ошибок → **designer**
- **3.2** Wireframes / структура экранов: форма ввода, превью/подтверждение, состояния загрузки и успеха/ошибки → **designer**
- **3.3** Визуальный стиль (цвета, типографика, отступы), спецификация компонентов и состояний; требования по доступности (a11y) и адаптивности → **designer**

**Результат:** макеты/спецификация UI, описание сценариев для разработчика и тестировщика.

---

## Этап 4. Реализация

- **4.1** Инициализация проекта: React (frontend), FastAPI (backend), структура папок по архитектуре → **developer**
- **4.2** Бекенд: API загрузки образца pptx, парсинг/извлечение стиля и структуры слайда-образца; хранение файла (временное или по правилам архитектора) → **developer**
- **4.3** Бекенд: приём текстового описания слайдов (JSON/формат по ТЗ), генерация pptx на основе описания и стиля образца (python-pptx или аналог) → **developer**
- **4.4** Бекенд: эндпоинт выдачи сгенерированного файла, валидация входных данных, обработка ошибок → **developer**
- **4.5** Фронтенд: форма ввода описания слайдов (заголовки, пункты по слайдам), загрузка файла-образца, отправка запроса на генерацию → **developer**
- **4.6** Фронтенд: отображение состояний (загрузка, прогресс, успех, ошибка), скачивание готового pptx → **developer**
- **4.7** Интеграция фронтенд–бекенд, проверка по критериям приёмки → **developer**

**Результат:** работающее приложение (локально).

---

## Этап 5. Тестирование

- **5.1** Тест-план и приоритеты покрытия (unit, integration, e2e) → **qa-test**
- **5.2** Unit-тесты бекенда: парсинг образца, генерация pptx, валидация → **qa-test**
- **5.3** Unit-тесты фронтенда: компоненты формы, обработка состояний → **qa-test**
- **5.4** Интеграционные тесты API (загрузка образца → описание → получение pptx) → **qa-test**
- **5.5** E2E-сценарии: полный сценарий от ввода данных до скачивания файла; граничные случаи (пустой ввод, неверный формат файла) → **qa-test**

**Результат:** набор тестов, отчёт о покрытии, список граничных случаев.

---

## Этап 6. Ревью кода и качество

- **6.1** Code review: читаемость, соответствие стандартам проекта, безопасность (обработка загружаемых файлов, валидация) → **code-review**
- **6.2** Рекомендации по производительности и исправление замечаний; при необходимости — итерация с developer → **code-review**
- **6.3** Исправление замечаний ревью → **developer** (повторный запуск по решению оркестратора)

**Результат:** пройденное ревью, внесённые правки.

---

## Этап 7. Документация

- **7.1** README проекта: назначение, запуск (frontend + backend), переменные окружения → **documentation**
- **7.2** Описание API (эндпоинты, форматы запроса/ответа, коды ошибок) → **documentation**
- **7.3** Актуализация ADR и онбординг-инструкций при необходимости → **documentation**

**Результат:** актуальная документация в репозитории.

---

## Этап 8. Безопасность

- **8.1** Аудит загрузки файлов (типы, размеры, проверка формата pptx), инъекции, секреты в коде и конфигурации → **security**
- **8.2** Рекомендации по исправлению уязвимостей; при необходимости — правки с developer → **security**
- **8.3** Внесение исправлений по результатам аудита → **developer** (повторный запуск по решению оркестратора)

**Результат:** отчёт/чек-лист безопасности, устранённые критические риски.

---

## Этап 9. Инфраструктура и деплой (DevOps)

- **9.1** Контейнеризация: Dockerfile для frontend и backend (или единый docker-compose) → **devops**
- **9.2** CI/CD: сборка, тесты, линтеры при коммите/PR; опционально — деплой в тестовое окружение → **devops**
- **9.3** Настройка окружения деплоя (хостинг/облако по выбору), переменные окружения, мониторинг базовый → **devops**
- **9.4** Деплой приложения и проверка доступности сервиса → **devops**

**Результат:** работающее приложение в целевом окружении, настроенный пайплайн.

---

## Этап 10. Координация и итерации (Оркестратор)

- **10.1** Разбиение плана на спринты/итерации, постановка задач агентам по этапам 1–9 → **orchestrator**
- **10.2** Передача контекста между агентами (требования → архитектор → дизайнер → developer и т.д.) → **orchestrator**
- **10.3** Решение о повторном запуске агентов после ревью, тестов или аудита безопасности (в т.ч. повторный запуск **developer** для исправления замечаний после этапов 6 и 8) → **orchestrator**
- **10.4** Отслеживание прогресса и зависимостей; координация финального приёмного тестирования и релиза → **orchestrator**

**Результат:** согласованный пайплайн от требований до деплоя, все этапы пройдены.

---

## Сводная последовательность агентов

1. **orchestrator** — запуск плана, постановка задач.
2. **business-analyst** — этап 1 (требования).
3. **architect** — этап 2 (архитектура).
4. **designer** — этап 3 (UI/UX).
5. **developer** — этап 4 (реализация).
6. **qa-test** — этап 5 (тесты).
7. **code-review** — этап 6 (ревью).
8. **documentation** — этап 7 (документация).
9. **security** — этап 8 (безопасность).
10. **devops** — этап 9 (инфраструктура и деплой).
11. **orchestrator** — сквозная координация и решение об итерациях (в т.ч. этап 10).

При необходимости после ревью или рефакторинга привлекается **refactoring** для улучшения структуры кода без смены поведения.
