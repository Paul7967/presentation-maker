# PRD / Техническое задание: Presentation Maker

**Версия:** 1.0  
**Дата:** 2025-03-06  
**Статус:** Для передачи архитектору (Этап 2)

---

## 1. Введение

**Presentation Maker** — веб-приложение для автоматической генерации презентаций в формате PPTX на основе текстового описания слайдов и загруженного образца слайда. Пользователь задаёт структуру презентации (заголовки и пункты по каждому слайду) и загружает файл-образец в формате PPTX; приложение создаёт готовый PPTX-файл, применяя к новому содержимому стиль и структуру образца.

**Стек:** фронтенд — React, бекенд — FastAPI.

**Целевая аудитория:** пользователи, которым нужно быстро получить презентацию в едином стиле без ручного копирования слайдов и форматирования (внутренние доклады, отчёты, типовые шаблоны).

---

## 2. Цели продукта

| Цель | Описание |
|------|----------|
| **Скорость** | Сократить время создания презентации за счёт ввода только текста и выбора одного образца слайда. |
| **Единообразие** | Обеспечить единый визуальный стиль и структуру слайдов на основе образца. |
| **Простота** | Минимизировать действия пользователя: описание слайдов + загрузка образца → получение файла. |
| **Предсказуемость** | Результат должен соответствовать описанию и визуально следовать образцу (шрифты, отступы, расположение заголовка и пунктов). |

---

## 3. Пользователи

| Роль | Описание | Ключевые потребности |
|------|----------|----------------------|
| **Автор презентации** | Сотрудник или фрилансер, создающий презентацию для доклада/отчёта. | Быстро получить PPTX в нужном стиле без ручного форматирования. |
| **Пользователь шаблона** | Тот, у кого уже есть корпоративный или персональный шаблон (один слайд в PPTX). | Применить этот шаблон к новому текстовому содержанию. |

В рамках MVP роли не различаются в системе (единый сценарий для одного типа пользователя).

---

## 4. Функциональные требования

### 4.1 Ввод данных

- **FR-1.** Пользователь может ввести текстовое описание слайдов: для каждого слайда — заголовок и список пунктов (одно- или многоуровневый список в рамках, определённых форматом ввода).
- **FR-2.** Пользователь может загрузить один файл в формате PPTX в качестве образца слайда (источник стиля и структуры).
- **FR-3.** Пользователь может инициировать генерацию презентации после ввода описания и загрузки образца.

### 4.2 Обработка

- **FR-4.** Система извлекает из образца PPTX стиль и структуру первого (или единственного) слайда: шрифты, размеры, выравнивание, отступы, расположение заголовка и области контента.
- **FR-5.** Система генерирует новый PPTX-файл: количество слайдов равно количеству слайдов в описании; содержимое каждого слайда (заголовок и пункты) берётся из описания; к каждому слайду применяется стиль/структура образца.

### 4.3 Результат

- **FR-6.** Пользователь может скачать сгенерированный PPTX-файл после успешной генерации.
- **FR-7.** При ошибке (валидация, обработка, сбой) пользователь получает понятное сообщение об ошибке.

### 4.4 Валидация и ограничения

- **FR-8.** Система проверяет формат загружаемого файла (PPTX), допустимый размер файла и допустимое количество слайдов в описании (см. раздел 6).
- **FR-9.** Система проверяет наличие и минимальную корректность текстового описания (не пустое, корректная структура).

---

## 5. Входные и выходные данные

### 5.1 Входные данные

| Вход | Формат / описание | Обязательность |
|------|-------------------|-----------------|
| **Текстовое описание слайдов** | Структурированные данные: массив слайдов; каждый слайд — заголовок (строка) и список пунктов (массив строк; уровень вложенности — по согласованию с архитектором, например один уровень для MVP). Передача: JSON в теле запроса или форма на фронтенде, преобразуемая в JSON. | Обязательно |
| **Образец слайда (PPTX)** | Файл с расширением `.pptx`, соответствующий формату Office Open XML. Содержит хотя бы один слайд. Используется первый слайд для извлечения стиля и структуры. | Обязательно |

### 5.2 Выходные данные

| Выход | Формат / описание |
|-------|-------------------|
| **Сгенерированная презентация** | Файл в формате PPTX. Количество слайдов = количеству слайдов в описании. Содержимое (заголовки и пункты) соответствует введённому описанию. Внешний вид и структура слайдов соответствуют образцу. |
| **Сообщение об ошибке** | Текстовое/структурированное сообщение (HTTP-код + тело ответа) с указанием причины (например: неверный формат файла, превышен размер, пустое описание, ошибка генерации). |

---

## 6. Ограничения

| Ограничение | Значение / описание |
|-------------|---------------------|
| **Формат образца** | Только PPTX (Office Open XML). Иные форматы (PPT, ODP и т.д.) не поддерживаются. |
| **Максимальный размер загружаемого файла** | Конкретное значение задаётся на этапе архитектуры (рекомендуемый порядок: 10–50 MB). |
| **Максимальное количество слайдов в описании** | Конкретное значение задаётся на этапе архитектуры (рекомендуемый порядок: от 1 до 100–200 слайдов для MVP). |
| **Минимальное количество слайдов** | Не менее 1 слайда в описании. |
| **Длина заголовка и пунктов** | Ограничения при необходимости задаются архитектором (например, максимальная длина строки или суммарный размер payload) для предотвращения злоупотреблений и сбоев. |
| **Одновременность** | В MVP допустимо последовательная обработка одного запроса на генерацию за раз (без обязательной очереди и фоновых задач). |

---

## 7. Граничные случаи и обработка ошибок

| Случай | Ожидаемое поведение |
|--------|---------------------|
| **Пустое описание слайдов** | Валидация не пройдена; сообщение типа «Введите описание хотя бы одного слайда». |
| **Ноль слайдов в описании** | Валидация не пройдена; сообщение о необходимости минимум одного слайда. |
| **Пустой заголовок или все пустые пункты у слайда** | Допустимо на усмотрение продукта: либо разрешать (слайд с пустым заголовком/пунктами), либо валидировать и возвращать ошибку. Рекомендация: разрешать пустые пункты; заголовок может быть пустой строкой или обязательным — зафиксировать в user stories. |
| **Файл не загружен** | Валидация не пройдена; сообщение «Загрузите образец в формате PPTX». |
| **Файл не PPTX** (расширение или MIME/сигнатура) | Ошибка валидации; сообщение «Поддерживается только формат PPTX». |
| **Повреждённый или нечитаемый PPTX** | Ошибка обработки; сообщение «Файл образца повреждён или не может быть прочитан». |
| **Образец без слайдов (пустая презентация)** | Ошибка обработки; сообщение «В файле образца не найден ни один слайд». |
| **Размер файла превышает лимит** | Ошибка валидации; сообщение «Превышен максимальный размер файла» (с указанием лимита). |
| **Количество слайдов в описании превышает лимит** | Валидация не пройдена; сообщение «Превышено максимальное количество слайдов» (с указанием лимита). |
| **Сбой при извлечении стиля из образца** | Ошибка обработки; сообщение о невозможности использовать образец (с возможностью fallback на дефолтный стиль — пост-MVP). |
| **Сбой при генерации PPTX** | Ошибка обработки; сообщение «Не удалось сгенерировать презентацию»; при возможности — логирование для диагностики. |
| **Таймаут запроса** | На этапе архитектуры задаётся таймаут; при превышении — ответ с кодом таймаута и понятным сообщением для пользователя. |

## 8. Нефункциональные требования (кратко)

- **Доступность:** веб-интерфейс доступен в современных браузерах (список уточняется дизайнером/архитектором).
- **Безопасность:** загрузка файлов — валидация типа и размера; не выполнять произвольный код из загружаемых файлов (обработка только как PPTX).
- **Производительность:** время ответа при генерации должно быть приемлемым для типовых объёмов (например, до 10–30 слайдов); точные целевые значения задаются на этапе архитектуры.

---

## 9. Вне области MVP (пост-MVP)

- Редактирование слайдов в браузере (WYSIWYG).
- Поддержка изображений/медиа в описании слайдов.
- Несколько образцов или выбор слайда-образца из загруженного файла.
- Сохранение проектов/шаблонов в аккаунте пользователя.
- Аутентификация и разграничение доступа.
- Очередь заданий и фоновые задачи для тяжёлых генераций.

---

## 10. Проверка полноты и непротиворечивости

- Входы (описание слайдов + образец PPTX) и выход (готовый PPTX) согласованы с целями продукта.
- Функциональные требования FR-1–FR-9 покрывают сценарий «ввод → генерация → скачивание» и валидацию.
- Ограничения и граничные случаи заданы однозначно; противоречий между разделами не выявлено.
- Документ готов к передаче архитектору для этапа 2 (архитектура и технический дизайн).
