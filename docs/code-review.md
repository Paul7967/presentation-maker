# Ревью кода — Presentation Maker

**Дата:** 2025-03-06  
**Этап:** 6. Ревью кода и качество (п. 6.1–6.2)  
**Объект:** backend (FastAPI), frontend (Vite + React). Эндпоинт POST /api/presentation/generate, загрузка файлов, валидация.

---

## 1. Общая оценка

Код в целом соответствует архитектуре и документации (api.md, architecture.md, ADR). Структура модулей понятная, валидация входных данных и слайдов реализована, проверка типа файла по MIME и сигнатуре OOXML присутствует, таймаут генерации (120 с) и лимиты из api.md соблюдаются. Обработка файлов выполняется в памяти (BytesIO), без записи во временные файлы на диск.

Выявлены: один **критический** момент по безопасности/производительности (возможность DoS при запросе без Content-Length и большом теле), несколько рекомендаций по стилю, читаемости и улучшению (в т.ч. порядок проверок, закрытие пула потоков, ключи списков во фронтенде). Критических багов в бизнес-логике не обнаружено.

## 2. Читаемость и стандарты

### Backend (Python)

- **Именование и структура:** Имена функций и переменных понятные (`_is_pptx_signature`, `_validate_slides`, `generate_presentation`). Модули разделены по слоям (api, services, models). Константы лимитов вынесены в `pptx_service` и переиспользуются в API — дублирование с `main.py` только для `MAX_REQUEST_BODY` (20 MB), что допустимо для middleware.
- **Комментарии:** Docstring'и есть у ключевых функций и эндпоинта; для сложной логики (например, проверка сигнатуры OOXML) комментарии уместны и присутствуют.
- **PEP 8 / Black / Ruff:** В проекте есть `pyproject.toml` с настройками black (line-length 100) и ruff (E, F, I; E501 отключён). Стиль кода в целом аккуратный; замечаний по отступам и длине строк не выявлено. Рекомендация: добавить в CI проверку `ruff check` и `black --check`, чтобы ревью опиралось на единый стандарт.
- **Мелкое:** В `app/models/schemas.py` определена Pydantic-модель `SlideInput`, но в эндпоинте валидация слайдов делается вручную через `_validate_slides()` и `json.loads(slides)`. Схемы не используются для входящего JSON — либо перейти на валидацию через Pydantic (единый стиль), либо кратко пояснить в комментарии, почему выбран ручной разбор (например, поле приходит как строка Form).

### Frontend (JS/React)

- **Именование и структура:** Компоненты и константы названы ясно (`SlideCard`, `API_GENERATE`, `MESSAGES`). Логика формы сосредоточена в `App.jsx`, компонент `SlideCard` принимает ограничения через пропсы — читаемо.
- **ESLint:** Используется flat config (`eslint.config.js`), включены recommended и react-hooks. Отдельного Prettier в проекте не видно; при желании унифицировать форматирование можно добавить Prettier и интеграцию с ESLint.
- **Рекомендации:**
  - В списках слайдов и пунктов используется `key={i}` / `key={index}`. Для стабильной идентификации при перестановке/удалении лучше использовать стабильный id (например, `slide.id` или `crypto.randomUUID()` при добавлении слайда), чтобы избежать лишних перерисовок и предупреждений React.
  - Вынести сообщения об ошибках и подсказки (сейчас частично в `constants.js`) в один модуль/объект для согласованности с бекендом (как в api.md).

## 3. Безопасность

### Загрузка файлов

- **Тип и размер:** Проверяется наличие файла, непустое имя; размер после чтения в память (`len(body) > MAX_FILE_SIZE` → 413); пустой файл → 400. MIME ограничен множеством `PPTX_MIMES` плюс явно разрешён `application/zip`. Сигнатура OOXML проверяется в `_is_pptx_signature()` (ZIP + наличие `ppt/` в первых 4096 байтах). Соответствует api.md и ADR 004.
- **Критично (см. раздел 5):** При запросе **без заголовка Content-Length** (например, chunked encoding) middleware `BodyLimitMiddleware` не срабатывает, а в эндпоинте весь файл читается через `await template.read()` без ограничения размера до чтения. Злоумышленник может отправить очень большое тело (например, десятки гигабайт) и вызвать исчерпание памяти на сервере (DoS). Необходимо ограничить размер читаемого тела на уровне приложения или ASGI-сервера.
- **Рекомендация:** Не полагаться только на Content-Length. Ограничить размер тела запроса в конфигурации uvicorn (или другого ASGI-сервера) и/или читать файл чанками с контролем суммарного размера и возвратом 413 при превышении до полной загрузки в память.

### Валидация входных данных (slides)

- **JSON и структура:** Наличие поля `slides`, парсинг через `json.loads`, проверка типа массива, проверка количества слайдов (1–100), типов полей (title — строка, items — массив строк), лимиты длины заголовка (500), пункта (2000), количества пунктов на слайд (50). Соответствует api.md. Риск переполнения или инъекций в рамках текущей логики снижен.
- **Выполнение кода из данных:** Загруженный файл обрабатывается только как PPTX (python-pptx). Макросы и произвольный код из файла не выполняются. Данные из `slides` используются только как строки для подстановки в текст плейсхолдеров. Критических рисков исполнения кода из входных данных не выявлено.

### Прочее

- CORS ограничен списком origins (localhost/127.0.0.1 для разработки). В production необходимо ограничить `allow_origins` доверенными доменами (как указано в api.md и ADR 004).

## 4. Производительность

- **Память:** Файл образца и сгенерированный PPTX держатся в памяти (BytesIO). Временные файлы на диск не создаются — очистка temp-файлов не требуется. При одновременной обработке нескольких больших запросов пиковое потребление памяти может быть высоким; лимит размера тела (после исправления из п. 5) и ограничение числа воркеров пула (`max_workers=2`) частично смягчают риск.
- **Таймаут:** Генерация выполняется в `ThreadPoolExecutor` с `asyncio.wait_for(..., timeout=GENERATE_TIMEOUT_SEC)` (120 с). При превышении возвращается 504 и сообщение из api.md — корректно.
- **Рекомендации:**
  - Явно закрывать `ThreadPoolExecutor` при остановке приложения (в lifespan FastAPI), чтобы не оставлять фоновые потоки.
  - Рассмотреть ограничение размера тела на уровне ASGI (uvicorn/гиперкорн) в дополнение к проверкам в приложении — это также защитит от DoS по памяти (см. раздел 5).

## 5. Критические замечания

1. **Ограничение размера тела при отсутствии Content-Length (безопасность/DoS).**  
   Сейчас лимит 20 MB проверяется только: (a) в middleware по заголовку `Content-Length`, (b) в эндпоинте после полного чтения файла в память. Если клиент отправляет запрос с `Transfer-Encoding: chunked` без `Content-Length`, middleware не срабатывает, а `template.read()` читает весь поток до конца. Это позволяет отправить запрос с телом произвольно большого размера и исчерпать память сервера.  
   **Рекомендация:** Ввести жёсткое ограничение размера тела запроса на уровне ASGI-сервера (например, uvicorn) и/или читать тело файла чанками с контролем суммарного размера и возвратом 413 при превышении лимита до полной загрузки в память.

## 6. Некритические улучшения

- **Backend:** Закрывать `ThreadPoolExecutor` в lifespan при shutdown (например, `_executor.shutdown(wait=True)`), чтобы корректно завершать потоки при остановке приложения.
- **Backend:** Рассмотреть использование Pydantic для валидации поля `slides` (после парсинга JSON в список) для единообразия с `schemas.py` и сокращения дублирования правил валидации.
- **Backend:** В api.md указаны только MIME `application/vnd...presentation` и `application/octet-stream`. Разрешённый в коде `application/zip` — осознанное расширение (pptx — ZIP). Имеет смысл отразить это в api.md или в комментарии в коде.
- **Frontend:** Использовать стабильные ключи для элементов списка слайдов и пунктов (например, уникальный id слайда), а не индекс в массиве.
- **Frontend:** Добавить таймаут или AbortController для `fetch(API_GENERATE, ...)`, чтобы при долгой генерации пользователь мог получить сообщение о таймауте (например, 120 с) вместо бесконечного ожидания, если бекенд не ответит.
- **Стандарты:** Включить в CI проверки ruff и black (backend), при желании — Prettier (frontend) для единообразного форматирования.

## 7. Исправления по ревью (п. 6.3)

- **Критическое замечание (раздел 5):** Реализовано чтение тела файла чанками (1 MB) с контролем суммарного размера; при превышении 20 MB возвращается 413 до полной загрузки в память. Запросы без `Content-Length` (chunked) больше не приводят к неограниченному потреблению памяти.
- **Некритическое:** В `lifespan` приложения добавлен вызов `shutdown_executor()` при остановке приложения для корректного завершения потоков `ThreadPoolExecutor`.
