**Связь:** [PRD](prd.md), [План разработки](dev_plan.md)

---

## 1. Обзор

Presentation Maker — веб-приложение с раздельными клиентом и сервером: пользователь вводит текстовое описание слайдов и загружает образец PPTX; сервер извлекает стиль из образца, генерирует новый PPTX и возвращает его для скачивания.

**Стек:**
- **Фронтенд:** React (SPA)
- **Бекенд:** FastAPI (Python)
- **Работа с PPTX:** python-pptx (чтение образца, генерация презентации)

## 3. Границы компонентов

### 3.1 Фронтенд (React)

- **Назначение:** UI для ввода описания слайдов, загрузки образца PPTX, запуска генерации и скачивания результата.
- **Ответственность:**
  - Валидация ввода на клиенте (формат, количество слайдов, размер файла в пределах лимита).
  - Отправка запроса на генерацию (multipart: файл + JSON с описанием слайдов).
  - Отображение состояний: загрузка, успех, ошибка; скачивание файла по ответу.
- **Не делает:** парсинг PPTX, генерацию файлов, хранение файлов — только вызов API и отображение.

### 3.2 Бекенд (FastAPI)

- **Назначение:** приём запросов, валидация, извлечение стиля из образца, генерация PPTX, отдача файла.
- **Ответственность:**
  - Валидация типа и размера загружаемого файла, формата описания слайдов.
  - Парсинг образца PPTX (первый слайд), извлечение стилей и структуры (шрифты, отступы, расположение).
  - Генерация нового PPTX по описанию с применением извлечённого стиля.
  - Возврат сгенерированного файла в ответе или сообщение об ошибке в структурированном виде.
- **Не делает:** постоянное хранение файлов пользователей (в MVP — временные файлы на время запроса или в temp с очисткой).

### 3.3 Взаимодействие

- Связь по **HTTP/HTTPS**. API — REST-подобный; единственный сценарий MVP — один эндпоинт генерации (загрузка образца + описание слайдов → получение PPTX).
- Контракт API описан в [docs/api.md](api.md).

## 4. Ключевые технические решения

### 4.1 API

- Один эндпоинт генерации: **POST /api/presentation/generate**.
- Тело запроса: **multipart/form-data** (файл образца + поле с JSON описания слайдов).
- Ответ: либо **application/vnd.openxmlformats-officedocument.presentationml.presentation** (файл), либо JSON с ошибкой.
- Подробно: [api.md](api.md), [ADR 002](adr/002-api.md).

### 4.2 Работа с PPTX

- Библиотека **python-pptx** для чтения образца и записи новой презентации.
- Из образца используется **первый слайд**: извлекаются стили текста (шрифт, размер, цвет), параметры placeholder’ов (заголовок, тело), отступы и при необходимости размеры слайда.
- Генерация: создаётся новая презентация, для каждого слайда из описания создаётся слайд с тем же layout’ом/мастером, заполняются заголовок и пункты с применением извлечённых стилей.
- Обоснование выбора библиотеки: [ADR 003](adr/003-pptx-lib.md).

### 4.3 Обработка файлов на бекенде

- Загруженный файл проверяется по **MIME-типу** и (опционально) по **сигнатуре** (ZIP/Office Open XML); расширение `.pptx` не является достаточной проверкой.
- Файл обрабатывается только как PPTX (парсинг через python-pptx); произвольный код из файла не выполняется.
- В MVP сгенерированный файл возвращается в теле ответа; временные файлы (если пишутся на диск) хранятся в выделенной директории с очисткой после ответа или по таймеру.

## 5. Нефункциональные требования (кратко)

- **Лимиты:** размер загружаемого файла — до **20 MB**; количество слайдов в описании — от 1 до **100** (MVP).
- **Таймаут:** запрос на генерацию — **120 секунд**; при превышении возвращается 504 с понятным сообщением.
- **Масштабируемость:** в MVP — один инстанс; при росте нагрузки возможны очередь заданий и отдельный воркер (пост-MVP).
- **Безопасность:** валидация типа и размера файла, запрет выполнения кода из загружаемых данных; при деплое — HTTPS, при необходимости лимиты по частоте запросов (пост-MVP).

Детали лимитов, кодов ошибок и рекомендаций по безопасности — в [api.md](api.md) и [ADR](adr/) (в т.ч. [ADR 004](adr/004-nfr-security.md)).
