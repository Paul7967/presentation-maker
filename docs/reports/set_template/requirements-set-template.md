# Требования: тип шаблона слайда (layout) и заметки докладчика

**Версия:** 1.0  
**Дата:** 2025-03-08  
**Связь:** [План внедрения (dev_plan_set_template.md)](../../plans/dev_plan_set_template.md), [API](../../api.md), [Архитектура](../../architecture.md)

---

## 1. Цель и контекст

**Цель:** добавить в приложение Presentation Maker поддержку (1) типа шаблона (layout) для каждого слайда и (2) заметок докладчика (speaker notes). Редактирование — в UI (режимы «по слайдам» и «через JSON») и в формате JSON. Бекенд (python-pptx): выбор layout при создании слайда, запись в notes_slide. Формат слайда: `{ title, items, layoutId?, notes? }`.

**Контекст:** в PPTX у слайдов можно указывать тип шаблона (layout из образца/темы) и заметки докладчика. Сейчас для всех слайдов применяется один layout первого слайда образца. Необходимо позволить задавать layout и заметки для каждого слайда.

---

## 2. Требования к двум фичам

### 2.1. Тип шаблона слайда (layout)

- Пользователь может задать для каждого слайда тип шаблона (layout) из образца PPTX.
- Тип шаблона идентифицируется через **layoutId**: индекс layout в образце (целое число) или имя layout (строка) — решение на этапе архитектуры.
- В режиме «По слайдам» — выбор типа шаблона через UI (выпадающий список после загрузки образца или универсальное поле ввода layoutId).
- В режиме «Через JSON» — поле `layoutId` в объекте слайда (опционально).
- В сгенерированном PPTX каждый слайд создаётся с выбранным layout; при невалидном или отсутствующем layoutId используется первый доступный layout образца (fallback).

### 2.2. Заметки докладчика (speaker notes)

- Пользователь может ввести для каждого слайда текст заметок докладчика.
- Заметки — строка произвольного текста (в пределах лимита длины); попадают в speaker notes итогового PPTX (notes_slide).
- В режиме «По слайдам» — поле ввода заметок (например, textarea) в карточке слайда.
- В режиме «Через JSON» — поле `notes` в объекте слайда (опционально, строка).
- В сгенерированном PPTX для каждого слайда создаётся/заполняется notes_slide; пустые заметки допускаются (пустая строка или отсутствие поля).

## 3. Граничные случаи

### 3.1. Неизвестный/несуществующий layoutId

- Если переданный layoutId не соответствует ни одному layout в образце (индекс вне диапазона, имя не найдено) — использовать **первый доступный layout** образца (fallback).
- Не считать это ошибкой валидации: запрос принимается, генерация выполняется с fallback-layout.

### 3.2. Пустые заметки

- Пустая строка `notes: ""` и отсутствие поля `notes` обрабатываются одинаково: в PPTX записывается пустой текст заметок или notes_slide не заполняется (в зависимости от реализации python-pptx).
- Валидация не требует обязательного наличия поля `notes`.

### 3.3. Лимит длины заметок

- Установить максимальную длину поля `notes` на один слайд (например, 2000 или 4000 символов по аналогии с пунктами). Конкретное значение — на этапе архитектуры.
- При превышении лимита — ошибка валидации на бекенде и фронтенде с сообщением о превышении лимита.

### 3.4. Переключение режимов (по слайдам ↔ через JSON) с новыми полями

- При переключении «По слайдам» → «Через JSON» в сериализованный JSON включаются поля `layoutId` и `notes` (если заданы).
- При переключении «Через JSON» → «По слайдам» парсинг и валидация учитывают опциональные `layoutId` и `notes`; при успехе state слайдов обновляется с сохранением этих полей.
- Данные не теряются при переключении туда-обратно при валидном вводе; консистентность между режимами обеспечивается единым форматом слайда `{ title, items, layoutId?, notes? }`.

### 3.5. Образец не загружен

- До загрузки образца список layout’ов неизвестен. Варианты: показывать выбор типа шаблона только после загрузки образца; либо универсальное поле «layoutId» (индекс или имя) с подсказкой — решение на этапе дизайна/архитектуры.

## 4. Критерии приёмки

| ID | Критерий приёмки | Приоритет |
|----|------------------|-----------|
| AC-1 | Пользователь может задать тип шаблона (layout) для каждого слайда в режиме «По слайдам» (выбор в UI). | Must |
| AC-2 | Пользователь может задать тип шаблона для каждого слайда в режиме «Через JSON» через поле `layoutId` в объекте слайда. | Must |
| AC-3 | Пользователь может ввести заметки докладчика для каждого слайда в режиме «По слайдам» (поле ввода в карточке слайда). | Must |
| AC-4 | Пользователь может ввести заметки докладчика в режиме «Через JSON» через поле `notes` в объекте слайда. | Must |
| AC-5 | JSON-формат описания слайдов поддерживает опциональные поля `layoutId` и `notes`; загрузка/сохранение JSON сохраняет эти поля. | Must |
| AC-6 | Сгенерированный PPTX содержит для каждого слайда выбранный layout (или первый при невалидном layoutId). | Must |
| AC-7 | Сгенерированный PPTX содержит заметки докладчика в notes_slide для каждого слайда, где notes заданы. | Must |
| AC-8 | При невалидном или отсутствующем layoutId используется первый доступный layout образца без ошибки валидации. | Must |
| AC-9 | При превышении лимита длины заметок показывается ошибка валидации на бекенде и фронтенде. | Must |

---

## 5. Уточнения для следующих этапов

- **Архитектор:** формат layoutId (индекс и/или имя), стратегия маппинга на layout образца, точный лимит длины notes.
- **Дизайнер:** размещение выбора типа шаблона и поля заметок в SlideCard; поведение до/после загрузки образца; a11y.
- **Разработчик:** валидация layoutId и notes на бекенде; выбор layout и запись notes_slide в pptx_service; расширение state и UI на фронтенде.
