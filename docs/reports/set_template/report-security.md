# Отчёт безопасности: поля layoutId и notes

**Дата:** 2025-03-08  
**Объект:** новые поля запроса API — layoutId и notes (план set_template).

---

## 1. Риски и меры

### 1.1. layoutId

- **Тип:** число (индекс) или строка (имя layout).
- **Инъекции:** значение не подставляется в команды ОС и не выполняется как код; используется только для выбора layout в python-pptx (индекс в списке или get_by_name). Риск инъекции кода отсутствует.
- **Длина строки:** на бекенде ограничена 200 символами (валидация в _validate_slides). Превышение возвращает 400 с сообщением «Превышен лимит: имя layout не более 200 символов.» — защита от переполнения и злоупотреблений.
- **Невалидное значение:** при индексе вне диапазона или неизвестном имени используется fallback (первый layout), ошибка валидации не возвращается — поведение безопасное и предсказуемое.

### 1.2. notes

- **Тип:** строка (текст заметок докладчика).
- **Инъекции:** текст записывается только в текст notes_slide через API python-pptx (text_frame.paragraphs). Не используется в SQL, командах ОС или разметке без экранирования на стороне сервера; в MVP сервер отдаёт бинарный PPTX, не встраивает notes в HTML. Риск XSS/инъекций при типичном использовании (скачивание PPTX) отсутствует.
- **Длина:** на бекенде ограничена 4000 символами (MAX_NOTES_LEN). Превышение возвращает 400 с сообщением о лимите заметок — защита от чрезмерного размера тела запроса и DoS через огромные строки.
- **Содержимое:** специальные символы и переносы строк допустимы; python-pptx записывает текст в XML-часть PPTX (экранирование выполняет библиотека).

### 1.3. Валидация на бекенде

- Все проверки выполняются в _validate_slides до генерации: тип layoutId (int или str), длина строки layoutId ≤ 200; тип notes (str), длина notes ≤ 4000. При нарушении возвращается 400 с понятным сообщением. Данные не попадают в генерацию без прохождения валидации.

---

## 2. Рекомендации

- Текущих мер достаточно для MVP: лимиты и типы проверяются на бекенде, инъекции и выполнение кода из layoutId/notes не применимы в данной архитектуре.
- При появлении сценариев, где содержимое notes отображается в веб-интерфейсе (например, предпросмотр), необходимо экранировать вывод для предотвращения XSS.

Риски полей layoutId и notes учтены: валидация типа и длины на бекенде, отсутствие использования в командах/коде. Дополнительные правки не требуются.
