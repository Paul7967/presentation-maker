# План внедрения: ввод/загрузка/сохранение JSON и переключение режимов ввода

**Цель:** добавить в приложение Presentation Maker возможность вводить описание презентации в формате JSON, загружать его из файла, сохранять в файл и переключаться между режимом «по слайдам» и режимом «через JSON».

**Контекст:** текущее приложение (React frontend, FastAPI backend, монорепо) уже передаёт описание слайдов на API в виде JSON ([docs/api.md](../api.md)). Бекенд менять не требуется — меняется только интерфейс ввода и способы получения/сохранения этих данных на клиенте.

**Связь:** [Архитектура](../architecture.md), [API](../api.md), [План разработки](dev_plan.md).

**Координация:** агент orchestrator ведёт выполнение плана с этапа 1: ставит задачи агентам, передаёт контекст между этапами и при необходимости запускает итерации после ревью или тестов. Этап 9 формально описывает эту сквозную роль и подведение итогов.

**Примечания по испольнению:** в нижней части отчета о проделаной работе каждого агента в чате добавлять строчку с указанием наименования агента который выполнил задачу

---

## Этап 1. Уточнение требований и критериев приёмки

- **1.1** Зафиксировать требования к четырём фичам: ввод JSON в интерфейсе, drag-and-drop загрузка .json, сохранение в .json, переключение режимов в хедере; граничные случаи (невалидный JSON, пустой файл, синхронизация данных при переключении режимов) → **business-analyst**
- **1.2** Сформулировать критерии приёмки и приоритеты (MVP набора: все четыре фичи в одном релизе) → **business-analyst**
- **1.3** Передать обновлённые требования оркестратору и архитектору; при необходимости обновить user stories / PRD → **business-analyst**

**Результат:** документ с требованиями к фичам (JSON-ввод, DnD, сохранение, переключение режимов), критерии приёмки, граничные случаи.

---

## Этап 2. Архитектура и контракт данных

- **2.1** Определить единый формат JSON на фронтенде (совместимый с полем `slides` в API): структура массива слайдов `[{ title, items }]`, лимиты по [api.md](../api.md) → **architect**
- **2.2** Решить стратегию синхронизации: один источник истины (slides state) с производным JSON-представлением при переключении в режим JSON, либо два представления с конвертацией при смене режима; поведение при невалидном JSON в текстовом поле или в загруженном файле → **architect**
- **2.3** Зафиксировать: не требуется изменений бекенда и API; при необходимости — краткое дополнение к docs/architecture.md (раздел про фронтенд) → **architect**

**Результат:** решение по формату и синхронизации данных, отсутствие изменений контракта API, при необходимости — обновление архитектурного документа.

---

## Этап 3. UI/UX режима JSON и переключения

- **3.1** User flow: переключение «По слайдам» ↔ «Через JSON», ввод/редактирование JSON в интерфейсе, drag-and-drop зона для .json, кнопка «Сохранить в JSON», отображение ошибок (невалидный JSON, лимиты) → **designer**
- **3.2** Размещение переключателя режимов в хедере (кнопки/табы), вид зоны ввода JSON (textarea или подсветка), вид DnD-зоны и кнопки сохранения; состояния ошибки и подсказки → **designer**
- **3.3** Доступность (a11y) и адаптивность новых элементов; согласованность с текущим визуальным стилем ([docs/design-spec.md](../design-spec.md), [docs/ui-spec.md](../ui-spec.md) при наличии) → **designer**

**Результат:** спецификация/макеты UI для режима JSON, DnD и переключателя; сценарии для разработчика и тестировщика.

---

## Этап 4. Реализация

- **4.1** Состояние приложения: режим ввода (`bySlides` | `byJson`), при необходимости — строка JSON для режима «через JSON»; переключатель в хедере, сохранение выбранного режима при переключении без потери данных → **developer**
- **4.2** Режим «По слайдам»: оставить текущую форму (SlideCard и т.д.) без изменений логики; при переключении в «Через JSON» подставлять в текстовое поле текущие слайды, сериализованные в JSON → **developer**
- **4.3** Режим «Через JSON»: текстовое поле (textarea) для ввода/редактирования JSON; валидация при вводе и при отправке формы (формат, лимиты по api.md); при переключении в «По слайдам» парсить JSON и заполнять слайды (с обработкой ошибок парсинга и лимитов) → **developer**
- **4.4** Загрузка из файла: зона drag-and-drop для файлов .json; при drop или выборе файла — чтение, парсинг, обновление данных (слайды или JSON-строка в зависимости от выбора синхронизации); проверка расширения и типа содержимого; сообщения об ошибках → **developer**
- **4.5** Сохранение в файл: кнопка «Сохранить в JSON» / «Экспорт в JSON», формирование содержимого из текущего состояния слайдов, скачивание файла (например, `presentation.json`) через Blob/URL без обращения к бекенду → **developer**
- **4.6** Интеграция: отправка на генерацию (POST /api/presentation/generate) в обоих режимах использует один и тот же массив слайдов (из state); повторная проверка валидации перед отправкой → **developer**

**Результат:** работающий в браузере функционал: переключение режимов, ввод JSON, DnD загрузка .json, сохранение в .json.

---

## Этап 5. Тестирование

- **5.1** Тест-план по новым сценариям: переключение режимов с сохранением данных, ввод валидного/невалидного JSON, загрузка .json через DnD и через выбор файла, сохранение в .json, генерация презентации из режима JSON → **qa-test**
- **5.2** Unit-тесты фронтенда: парсинг/сериализация JSON, валидация лимитов, обработка ошибок при загрузке файла → **qa-test**
- **5.3** Интеграционные/E2E сценарии: полный цикл «переключение в JSON → ввод/загрузка → сохранение в файл → переключение в слайды → генерация»; граничные случаи (пустой JSON, битый JSON, слишком большой массив) → **qa-test**

**Результат:** набор тестов для новых фич, отчёт о покрытии, проверка граничных случаев.

---

## Этап 6. Ревью кода

- **6.1** Code review изменений: читаемость, соответствие стилю проекта и линтерам, безопасность (парсинг пользовательского JSON без eval, ограничение размера загружаемого файла при необходимости) → **code-review**
- **6.2** Рекомендации по исправлениям; при необходимости — итерация с **developer** (по решению оркестратора) → **code-review**

**Результат:** пройденное ревью, внесённые правки.

---

## Этап 7. Документация

- **7.1** Обновить README (и при необходимости пользовательскую документацию): описание режимов ввода «по слайдам» и «через JSON», загрузка и сохранение JSON-файла → **documentation**
- **7.2** При необходимости кратко отразить в docs/architecture.md или docs/api.md поддержку двух режимов ввода на фронтенде (без изменения контракта API) → **documentation**

**Результат:** актуальная документация по новому функционалу.

---

## Этап 8. Безопасность (по усмотрению оркестратора)

- **8.1** Проверить риски: загрузка пользователем .json файла (ограничение размера, парсинг без eval, отсутствие выполнения кода); отсутствие передачи JSON-файла на бекенд (файл обрабатывается только на клиенте) → **security**
- **8.2** При выявлении рисков — рекомендации и при необходимости правки с **developer** → **security**

**Результат:** учёт рисков загрузки/парсинга JSON на клиенте, при необходимости — правки.

---

## Этап 9. Координация и итерации

- **9.1** Постановка задач агентам по этапам 1–8, передача контекста (требования → архитектор → дизайнер → developer и т.д.) → **orchestrator**
- **9.2** Решение о необходимости этапа безопасности (8) и о повторном запуске developer после ревью или тестов → **orchestrator**
- **9.3** Отслеживание зависимостей (дизайн до реализации, реализация до тестов и ревью) и финальная приёмка → **orchestrator**

**Результат:** согласованное внедрение всех четырёх фич, все этапы выполнены.
