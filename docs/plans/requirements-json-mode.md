# Требования: режим ввода JSON в Presentation Maker

**Версия:** 1.0  
**Дата:** 2025-03-08  
**Связь:** [План разработки (Этап 1)](dev_plan_2.md), [API](api.md), [Архитектура](architecture.md)

---

## 1. Цель и контекст

**Цель:** дать пользователю возможность вводить описание презентации в формате JSON, загружать его из файла, сохранять в файл и переключаться между режимом «по слайдам» и режимом «через JSON». Изменения только на фронтенде (React); бекенд и контракт API не меняются.

**Контракт API:** поле `slides` — строка UTF-8 с JSON-массивом слайдов вида `[{ "title": "...", "items": ["...", ...] }]`. Лимиты по [api.md](api.md):

| Параметр | Значение |
|----------|----------|
| Количество слайдов | 1–100 |
| Длина заголовка слайда | до 500 символов |
| Длина одного пункта | до 2000 символов |
| Пунктов на слайд | до 50 |

## 2. Требования к четырём фичам

### 2.1. Ввод текста презентации в формате JSON в интерфейсе

- Пользователь может вводить и редактировать текст в формате JSON в специальном поле (например, textarea) в режиме «Через JSON».
- Поддерживаемый формат совместим с полем `slides` API: массив объектов с полями `title` (строка) и `items` (массив строк).
- При вводе возможна подсветка синтаксиса (опционально для MVP) и/или отображение ошибок валидации (невалидный JSON, нарушение лимитов).
- Валидация выполняется при попытке применить данные (переключение в режим «По слайдам», отправка на генерацию) и при желании — при вводе (debounce) для ранней обратной связи.

### 2.2. Загрузка текста из файла .json (drag-and-drop)

- Пользователь может загрузить файл с расширением `.json` путём перетаскивания (drag-and-drop) в выделенную зону или через выбор файла (click-to-browse).
- После загрузки содержимое файла читается на клиенте, парсится как JSON и используется для обновления данных презентации (слайды или JSON-строка в зависимости от выбранной архитектуры синхронизации).
- Проверяется расширение файла и при необходимости тип содержимого; при неверном формате или ошибке парсинга пользователю показывается понятное сообщение об ошибке.
- Обработка файла выполняется только на клиенте; файл не отправляется на бекенд.

### 2.3. Сохранение текста презентации в файл .json

- Пользователь может сохранить текущее описание слайдов в файл формата JSON (например, кнопка «Сохранить в JSON» / «Экспорт в JSON» в интерфейсе).
- Содержимое формируется из актуального состояния слайдов в приложении (единый источник истины) и скачивается как файл (например, `presentation.json`) через механизм Blob/URL без обращения к бекенду.
- Формат сохраняемого JSON совместим с полем `slides` API: массив `[{ "title", "items" }]`.

### 2.4. Переключение между режимами «по слайдам» и «через JSON» в хедере

- В хедере приложения размещены элементы переключения режима ввода: «По слайдам» и «Через JSON» (кнопки, табы или аналогичные контролы).
- При переключении в режим «Через JSON» в текстовое поле подставляется текущее представление слайдов в формате JSON (сериализация из состояния).
- При переключении в режим «По слайдам» данные из JSON-поля парсятся и записываются в состояние слайдов; при невалидном JSON или нарушении лимитов отображается ошибка, переключение может быть заблокировано или откатано с сохранением предыдущих данных.
- Данные не теряются при переключении туда-обратно при условии валидного ввода; консистентность между режимами обеспечивается единым источником истины (см. граничные случаи).

## 3. Критерии приёмки и приоритеты

**Приоритет:** MVP — все четыре фичи входят в один релиз.

| ID | Критерий приёмки | Приоритет |
|----|------------------|-----------|
| AC-1 | В режиме «Через JSON» пользователь видит поле ввода и может редактировать JSON в формате массива `[{ title, items }]`. | Must |
| AC-2 | При переключении «По слайдам» → «Через JSON» в поле отображается текущая презентация в виде JSON. | Must |
| AC-3 | При переключении «Через JSON» → «По слайдам» валидный JSON преобразуется в слайды и отображается в форме по слайдам. | Must |
| AC-4 | Есть зона drag-and-drop для загрузки файла .json; при drop или выборе файла содержимое парсится и обновляет данные презентации. | Must |
| AC-5 | Есть кнопка сохранения (экспорта) в .json; по нажатию скачивается файл с актуальными слайдами в формате API. | Must |
| AC-6 | В хедере есть переключатель режимов «По слайдам» / «Через JSON», переключение возможно в любой момент. | Must |
| AC-7 | При невалидном JSON (в поле или в файле) пользователь видит сообщение об ошибке без потери ранее введённых валидных данных. | Must |
| AC-8 | При отправке на генерацию (POST /api/presentation/generate) в обоих режимах используется один и тот же массив слайдов; перед отправкой выполняется проверка лимитов API. | Must |
| AC-9 | Подсветка синтаксиса JSON в поле ввода — опционально (Nice to have). | Should |

## 4. Граничные случаи

### 4.1. Невалидный JSON

- **В текстовом поле (режим «Через JSON»):** при переключении в «По слайдам» или при нажатии «Сгенерировать» выполнить парсинг; при синтаксической ошибке показать сообщение (например, «Некорректный JSON: …») и не обновлять слайды / не отправлять запрос. Текущие валидные данные (последний успешный парс или состояние слайдов) сохраняются.
- **В загруженном файле:** при парсинге ошибки — показать сообщение («Файл не является корректным JSON» или аналогичное), не менять текущие слайды/JSON.

### 4.2. Пустой файл или пустой массив

- **Пустой файл .json:** считать невалидным (нет валидного JSON); показать ошибку, не менять данные.
- **Валидный JSON, но пустой массив `[]`:** не допускается API (минимум 1 слайд). При применении таких данных показать ошибку («Введите хотя бы один слайд») и не подменять текущие слайды пустым массивом.

### 4.3. Нарушение лимитов API

- При парсинге JSON проверить: количество слайдов 1–100, длина каждого `title` ≤ 500 символов, длина каждого элемента в `items` ≤ 2000 символов, длина `items` ≤ 50 на слайд.
- При нарушении — сообщение с указанием конкретного ограничения; данные можно не применять или применить с обрезкой (решение уточняется архитектором; по умолчанию — не применять и показать ошибку).

### 4.4. Синхронизация данных при переключении режимов

- **Источник истины:** один — состояние слайдов (массив объектов `{ title, items }`). Режим «Через JSON» показывает производное представление (сериализованный JSON); при переключении из JSON в слайды парсинг обновляет этот единственный источник.
- **Консистентность:** при переключении «По слайдам» → «Через JSON» всегда подставлять актуальные слайды в виде JSON. При переключении «Через JSON» → «По слайдам» только успешный парс и прохождение валидации лимитов обновляют слайды; иначе остаётся предыдущее состояние, пользователь видит ошибку.
- **Потеря данных:** не допускать потери введённых данных при переключении при валидном вводе; при невалидном — не перезаписывать слайды, сохранять последнее валидное состояние.

### 4.5. Дополнительные граничные случаи

- **Очень большой файл .json:** ограничить размер загружаемого файла на клиенте (например, 1–2 MB), чтобы избежать подвисания при парсинге; при превышении — сообщение «Файл слишком большой».
- **Неожиданная структура JSON:** если корень не массив или элементы без `title`/`items` — считать невалидным; сообщение «Некорректный формат описания слайдов» и не применять.
- **Кодировка файла:** ожидается UTF-8; при проблемах чтения — сообщение об ошибке.

## 5. User stories и уточнения для команды

### 5.1. Краткие user stories

- **US-1:** Как пользователь, я хочу переключаться между вводом по слайдам и вводом через JSON, чтобы использовать удобный мне способ редактирования.
- **US-2:** Как пользователь, я хочу вставить или отредактировать JSON в текстовом поле, чтобы быстро править структуру презентации.
- **US-3:** Как пользователь, я хочу перетащить файл .json в зону загрузки, чтобы подставить готовое описание слайдов из файла.
- **US-4:** Как пользователь, я хочу сохранить текущую презентацию в файл .json, чтобы использовать её позже или в другом месте.

### 5.2. Уточнения для архитектора

- Зафиксировать единый формат JSON на фронте (совместимый с `slides` в API) и стратегию синхронизации: один источник истины (slides state) с производным JSON при режиме «Через JSON», либо два представления с явной конвертацией при смене режима.
- Определить поведение при невалидном JSON в поле и в загруженном файле: только сообщение об ошибке или также частичное применение (например, валидные слайды) — в документе принято «не применять, показать ошибку».
- Рекомендация: ограничение размера загружаемого .json файла (например, 1 MB) на клиенте.

### 5.3. Уточнения для дизайнера

- Размещение переключателя режимов в хедере (кнопки/табы), визуальное выделение активного режима.
- Вид зоны ввода JSON: textarea с возможной подсветкой синтаксиса (опционально для MVP).
- Вид зоны drag-and-drop для .json и кнопки «Сохранить в JSON»; состояния hover, error, пустое состояние.
- Тексты сообщений об ошибках: невалидный JSON, пустой файл, нарушение лимитов (количество слайдов, длина заголовка/пункта, количество пунктов) — формулировки, согласованные с api.md.
