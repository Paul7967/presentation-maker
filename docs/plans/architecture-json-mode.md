## 1. Цель документа

Зафиксировать единый формат JSON на фронтенде, стратегию синхронизации данных между режимами «По слайдам» и «Через JSON», поведение при невалидном JSON и явное решение об отсутствии изменений бекенда и API.

---

## 2. Единый формат JSON на фронтенде

### 2.1. Структура

На фронтенде используется **единственный формат** описания слайдов, **полностью совместимый** с полем `slides` в API:

- **Тип:** массив объектов.
- **Каждый элемент:** объект с полями:
  - `title` — строка (заголовок слайда, может быть пустой).
  - `items` — массив строк (пункты списка на слайде; пустой массив допускается).

**Пример:**

```json
[
  { "title": "Заголовок первого слайда", "items": ["Пункт 1", "Пункт 2"] },
  { "title": "Второй слайд", "items": [] }
]
```

При отправке на генерацию (POST /api/presentation/generate) этот же массив сериализуется в строку и передаётся в поле `slides` без преобразований. Дополнительных полей или обёрток на фронте не вводится.

### 2.2. Лимиты (по api.md)

Валидные данные должны удовлетворять ограничениям API:

| Параметр | Значение |
|----------|----------|
| Количество слайдов | от **1** до **100** включительно |
| Длина заголовка слайда (`title`) | до **500** символов |
| Длина одного пункта (элемент `items`) | до **2000** символов |
| Количество пунктов на слайд (`items.length`) | до **50** |

Проверка этих лимитов выполняется на фронтенде при применении данных (переключение в «По слайдам», загрузка файла, отправка на генерацию). Пустой массив слайдов `[]` не допускается.

## 3. Стратегия синхронизации данных

### 3.1. Один источник истины

**Источник истины — состояние слайдов (slides state)** в React: массив объектов `{ title, items }`. Все отображения и действия опираются на него.

- **Режим «По слайдам»:** UI (форма, карточки слайдов) отображает и редактирует этот массив напрямую.
- **Режим «Через JSON»:** текстовое поле показывает **производное** представление — результат сериализации текущего `slides` в JSON-строку. Само поле не является хранилищем данных; при переключении в режим «Через JSON» содержимое поля **всегда формируется из slides state**.

### 3.2. Переключение «По слайдам» → «Через JSON»

- Действие: пользователь переключается в режим «Через JSON».
- Поведение: из текущего `slides` state выполняется **сериализация в JSON** (например, `JSON.stringify(slides, null, 2)` для читаемости) и результат подставляется в текстовое поле.
- Состояние слайдов не меняется; поле отражает актуальные данные.

### 3.3. Переключение «Через JSON» → «По слайдам»

- Действие: пользователь переключается в режим «По слайдам».
- Поведение:
  1. Содержимое текстового поля **парсится** как JSON.
  2. Выполняется **валидация** (структура + лимиты по п. 2.2).
  3. **При успешной валидации:** результат парсинга записывается в `slides` state, переключение завершается, форма по слайдам отображает новые данные.
  4. **При ошибке парсинга или валидации:** `slides` state **не обновляется**, пользователю показывается сообщение об ошибке; режим можно оставить «Через JSON» или повторно ввести данные. Предыдущее валидное состояние слайдов сохраняется.

### 3.4. Загрузка из файла .json

- В режиме «Через JSON» или «По слайдам»: после выбора/ drop файла содержимое читается и парсится как JSON.
- При успешном парсинге и прохождении валидации структуры и лимитов: данные записываются в **slides state**. В режиме «Через JSON» текстовое поле затем можно обновить из обновлённого state (подставить новый JSON).
- При ошибке: слайды не меняются, показывается сообщение об ошибке (см. раздел 4).

### 3.5. Сохранение в файл .json

- Содержимое файла формируется **только из slides state**: `JSON.stringify(slides, null, 2)` (или без отступов — по решению реализации). Никакой «отложенный» текст из поля ввода не используется — только актуальный массив слайдов.

### 3.6. Отправка на генерацию

- В обоих режимах запрос на POST /api/presentation/generate использует **один и тот же** массив слайдов из state. Перед отправкой выполняется повторная проверка валидации (в т.ч. лимиты). Строка для поля `slides` формируется сериализацией этого массива.

## 4. Поведение при невалидном JSON

### 4.1. Общие принципы

- **Парсинг:** только **JSON.parse()**. Использование `eval()` или других способов выполнения кода запрещено.
- **При любой ошибке:** данные **не применяются** к slides state; пользователю показывается сообщение об ошибке; текущие слайды остаются без изменений.

### 4.2. Валидация после парсинга

После успешного `JSON.parse()` выполняется проверка:

1. **Тип корня:** значение должно быть **массивом**. Иначе — ошибка «Некорректный формат описания слайдов» (или аналог).
2. **Структура каждого элемента:** каждый элемент массива — объект с полями **title** (строка) и **items** (массив строк). Отсутствие поля, неверный тип (например, `title` — число) — ошибка формата.
3. **Лимиты (см. п. 2.2):**
   - количество слайдов: 1–100;
   - для каждого слайда: `title.length ≤ 500`, каждый элемент `items` — строка длиной ≤ 2000, `items.length ≤ 50`.

При первом нарушении возвращается понятное сообщение (например, «Превышено максимальное количество слайдов (100)», «Превышена длина заголовка слайда (500 символов)») и данные не применяются.

### 4.3. Невалидный JSON в текстовом поле

- **Синтаксическая ошибка JSON** (например, лишняя запятая, не закрытая скобка): при переключении в «По слайдам» или при нажатии «Сгенерировать» — вызов `JSON.parse()` бросает исключение. Поймать его, показать сообщение вида «Некорректный JSON: &lt;причина&gt;» (или кратко «Некорректный JSON»), не обновлять slides.
- **Валидный JSON, но не прошедший проверку структуры/лимитов:** сообщение с указанием конкретного нарушения (см. 4.2), slides не обновлять.

### 4.4. Невалидный JSON в загруженном файле

- **Ошибка парсинга** (файл не JSON): сообщение «Файл не является корректным JSON» (или аналог), текущие слайды не менять.
- **Пустой файл:** трактовать как невалидный JSON — сообщение об ошибке, данные не применять.
- **Валидный JSON, но не прошедший структуру/лимиты:** то же, что в 4.2 — сообщение с указанием причины, данные не применять.
- **Рекомендация (из требований):** ограничить размер загружаемого файла на клиенте (например, 1–2 MB); при превышении — сообщение «Файл слишком большой», парсинг не выполнять.

Дополнительные граничные случаи (очень большие файлы, неожиданная структура, кодировка UTF-8) учтены в [requirements-json-mode.md](requirements-json-mode.md); реализация должна им следовать.

## 5. Отсутствие изменений бекенда и API

**Решение зафиксировано:** в рамках внедрения режима «Через JSON» и связанных фич (ввод/загрузка/сохранение JSON, переключение режимов) **бекенд и API не меняются**.

- Контракт **POST /api/presentation/generate** остаётся по [api.md](api.md): multipart/form-data, поля `template` и `slides` (строка UTF-8 с JSON-массивом `[{ "title", "items" }]`), те же лимиты и коды ответов.
- Валидация на бекенде не меняется; фронтенд по-прежнему отправляет поле `slides` в том же формате.
- Все изменения касаются **только фронтенда**: два режима ввода, единый формат JSON в state, производное отображение в текстовом поле, парсинг/валидация на клиенте, загрузка и сохранение .json файлов на клиенте без участия сервера.
