# План внедрения: тип шаблона слайда (layout) и заметки докладчика

**Цель:** добавить в приложение Presentation Maker поддержку типа шаблона (layout) для каждого слайда и заметок для докладчика (speaker notes), с возможностью задания и редактирования в UI (режимы «по слайдам» и «через JSON») и в формате JSON.

**Контекст:** в PPTX у слайдов можно указывать тип шаблона (layout из образца/темы) и заметки докладчика. Текущее приложение (React frontend, FastAPI backend, монорепо) передаёт на API массив слайдов с полями `title` и `items`. Бекенд использует python-pptx и для всех слайдов применяет один и тот же layout первого слайда образца. Необходимо: (1) для каждого слайда позволять указывать тип шаблона и маппить его на layout из образца; (2) поддерживать заметки докладчика и записывать их в итоговый PPTX.

**Связь:** [Архитектура](../architecture.md), [API](../api.md), [План разработки](dev_plan.md), [План JSON-режима](dev_plan_2.md).

**Координация:** агент **orchestrator** ведёт выполнение плана с этапа 1: ставит задачи агентам, передаёт контекст между этапами и при необходимости запускает итерации после ревью или тестов. Этап 10 формально описывает эту сквозную роль и подведение итогов.

**Примечание по отчётам:** отчёты агентов о проделанной работе сохраняются в папку **docs/reports/set_template** (при необходимости создай её). В конце каждого отчёта указывать: «Выполнено агентом: &lt;имя агента&gt;».

---

## Этап 1. Уточнение требований и критериев приёмки

- **1.1** Зафиксировать требования к двум фичам: (1) тип шаблона слайда — выбор layout из образца для каждого слайда, отображение в итоговом pptx; (2) заметки докладчика — текст на слайд, попадающий в speaker notes итогового pptx → **business-analyst**
- **1.2** Граничные случаи: неизвестный/несуществующий layoutId в образце (fallback на первый layout), пустые заметки, лимит длины заметок (по аналогии с пунктами), поведение при переключении режимов (по слайдам ↔ JSON) с новыми полями → **business-analyst**
- **1.3** Критерии приёмки: пользователь может задать тип шаблона и заметки в обоих режимах ввода; JSON поддерживает поля `layoutId` и `notes`; сгенерированный pptx содержит выбранные layout’ы и заметки → **business-analyst**
- **1.4** Передать требования оркестратору и архитектору; при необходимости обновить user stories / PRD → **business-analyst**

**Результат:** документ с требованиями к фичам (тип шаблона, заметки), критерии приёмки, граничные случаи.

---

## Этап 2. Архитектура и контракт данных

- **2.1** Определить формат данных слайда в API и JSON: расширение структуры до `{ title, items, layoutId?, notes? }`; тип и допустимые значения `layoutId` (индекс layout в образце или имя/идентификатор — решение с учётом того, что в python-pptx layout доступен по индексу и по имени через `slide_layouts.get_by_name(name)`) → **architect**
- **2.2** Стратегия маппинга «тип шаблона» → layout образца: извлечение списка layout’ов из образца (имена и/или индексы), выбор по layoutId из запроса; fallback при невалидном layoutId (например, первый layout в образце) → **architect**
- **2.3** Лимиты: максимальная длина поля `notes` на слайд (например, 2000 или 4000 символов по аналогии с пунктами); валидация на бекенде и фронтенде → **architect**
- **2.4** Зафиксировать изменения контракта API в docs/api.md и при необходимости docs/architecture.md → **architect**

**Результат:** решение по формату слайда (layoutId, notes), стратегия маппинга layout, лимиты, обновление документации API/архитектуры.

## Этап 3. UI/UX выбора типа шаблона и ввода заметок

- **3.1** User flow: в режиме «По слайдам» — для каждого слайда выбор типа шаблона (выпадающий список или селект; список layout’ов неизвестен до загрузки образца — варианты: показывать после загрузки образца, либо универсальное поле «ID/индекс» с подсказкой) и поле ввода заметок докладчика (textarea) → **designer**
- **3.2** В режиме «Через JSON» — описание формата в подсказке и примере (поля `layoutId`, `notes`), отображение ошибок валидации новых полей → **designer**
- **3.3** Размещение элементов в карточке слайда (SlideCard): заголовок, пункты, тип шаблона, заметки; доступность (a11y) и согласованность с текущим стилем → **designer**
- **3.4** Поведение при отсутствии загруженного образца: как показывать выбор layout (например, свободное поле layoutId по индексу или отложенный выбор после загрузки) → **designer**

**Результат:** спецификация/макеты UI для типа шаблона и заметок в обоих режимах; сценарии для разработчика и тестировщика.

## Этап 4. Бекенд: валидация и генерация

- **4.1** Расширить валидацию поля `slides` в API: опциональные поля `layoutId` (число — индекс — и/или строка — имя layout, по решению этапа 2), `notes` (строка, лимит длины); нормализация (пустая строка notes → отсутствие поля или пустая строка) → **developer**
- **4.2** В сервисе генерации PPTX (pptx_service): при создании каждого слайда выбирать layout по layoutId из данных слайда (индекс или имя); при невалидном/отсутствующем layoutId использовать первый доступный layout образца; создавать слайд через `prs.slides.add_slide(chosen_layout)` → **developer**
- **4.3** Добавление заметок: для каждого слайда после заполнения контента обращаться к `slide.notes_slide` (python-pptx создаёт notes при первом обращении), записывать текст в `notes_slide.notes_placeholder.text_frame` или аналог; обрабатывать случай отсутствия notes placeholder в образце → **developer**
- **4.4** Извлечение списка layout’ов из образца: при необходимости — вспомогательная функция или эндпоинт для получения перечня layout’ов (имена/индексы) для UI; по решению архитектора и дизайнера может быть отложено (универсальный ввод layoutId без предзагрузки списка) → **developer**

**Результат:** API принимает и валидирует `layoutId` и `notes`; генерация pptx использует выбранный layout и записывает заметки докладчика.

---

## Этап 5. Фронтенд: состояние, UI и JSON

- **5.1** Расширить структуру слайда в state: `{ title, items, layoutId?, notes? }`; начальные значения при добавлении слайда (например, `layoutId: undefined` или `0`, `notes: ""`) → **developer**
- **5.2** Режим «По слайдам»: в SlideCard добавить поле выбора типа шаблона (по спецификации этапа 3) и поле ввода заметок; передача layoutId и notes в onChange/updateSlide; валидация лимита длины заметок на клиенте → **developer**
- **5.3** Режим «Через JSON»: при сериализации/парсинге включать `layoutId` и `notes`; обновить parseAndValidateSlides (и константы лимитов) для опциональных полей layoutId и notes, проверка лимита notes → **developer**
- **5.4** При отправке на генерацию (buildSlidesForApi / slidesForApi) передавать layoutId и notes в payload; синхронизация при переключении режимов без потери новых полей → **developer**
- **5.5** Загрузка/сохранение JSON-файла: сохранение и загрузка полей layoutId и notes без потери данных → **developer**

**Результат:** в UI можно задавать тип шаблона и заметки в обоих режимах; JSON-формат поддерживает layoutId и notes; данные корректно уходят на API.

## Этап 6. Тестирование

- **6.1** Тест-план: выбор layout по индексу/имени, fallback при невалидном layoutId; запись заметок в pptx; валидация лимитов notes на бекенде и фронтенде; переключение режимов с сохранением layoutId и notes → **qa-test**
- **6.2** Unit-тесты бекенда: валидация slides с layoutId и notes, генерация с разными layout’ами и заметками, граничные случаи (пустой layoutId, очень длинные notes) → **qa-test**
- **6.3** Unit-тесты фронтенда: парсинг/сериализация слайдов с layoutId и notes, валидация в slidesValidation → **qa-test**
- **6.4** Интеграционные/E2E: полный цикл с выбором типа шаблона и вводом заметок, генерация, проверка содержимого pptx (при возможности — проверка layout и notes в файле) → **qa-test**

**Результат:** набор тестов для новых полей и сценариев, отчёт о покрытии.
