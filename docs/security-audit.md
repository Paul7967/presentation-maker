# Аудит безопасности Presentation Maker

**Дата:** 2025-03-07  
**Область:** загрузка файлов, инъекции, секреты в коде и конфигурации (backend, frontend, конфиги).

---

## 1. Загрузка файлов

### 1.1 Backend (эндпоинт `/api/presentation/generate`)

| Проверка | Статус | Описание |
|----------|--------|----------|
| Лимит размера | ✅ | Лимит 20 MB: проверка по `Content-Length` в `BodyLimitMiddleware` (main.py) и по факту при чтении чанками в presentation.py (`MAX_FILE_SIZE`). Запросы без Content-Length также ограничиваются при потоковом чтении. |
| Тип файла (MIME) | ✅ | Проверка `content_type`: допускаются `application/vnd.openxmlformats-officedocument.presentationml.presentation`, `application/octet-stream`, при наличии — `application/zip`. |
| Сигнатура (реальный формат) | ✅ | Функция `_is_pptx_signature(content)`: проверка PK (ZIP) и наличия `ppt/` в первых 4096 байтах (OOXML). Файл с подходящим MIME, но не PPTX, отклоняется. |
| Расширение | ⚠️ | По расширению не проверяется (полагаемся на MIME и сигнатуру). Достаточно для безопасности; при желании можно добавить проверку `filename.lower().endswith('.pptx')` для сообщений об ошибках. |
| Временные файлы | ✅ | Файл читается в память (BytesIO), на диск не сохраняется — риск переполнения диска и оставшихся файлов отсутствует. |
| Исполняемый контент | ✅ | Обработка только через python-pptx (чтение/запись XML). Макросы и OLE не выполняются. |

**Вывод:** Загрузка файлов реализована с учётом лимита размера, проверки формата по MIME и сигнатуре; исполняемый контент не выполняется. Критических уязвимостей не выявлено.

### 1.2 Frontend

| Проверка | Статус | Описание |
|----------|--------|----------|
| Ограничение типа в UI | ✅ | `accept=".pptx"` у `<input type="file">`, проверка по расширению (`f.name.toLowerCase().endsWith(".pptx")`) и по размеру (`MAX_FILE_SIZE`) до отправки. |
| Размер до отправки | ✅ | Проверка `f.size > MAX_FILE_SIZE` (20 MB) с сообщением пользователю. |

**Вывод:** Фронтенд снижает нагрузку на сервер и улучшает UX; итоговая защита обеспечивается бекендом.

## 6. Рекомендации по исправлению / усилению

1. **CORS в production:** читать список разрешённых origin’ов из переменной окружения (например, `CORS_ORIGINS`) и при её отсутствии использовать текущий список для разработки. Это снизит риск случайного открытия API с неподходящего домена.
2. **Расширение файла (опционально):** при желании на backend можно дополнительно проверять `filename.lower().endswith('.pptx')` и при несоответствии возвращать 400 с сообщением «Поддерживается только формат PPTX» для единообразия с фронтендом.
3. **Документация:** в README уже указано про переменные окружения и ограничение CORS в production; при введении `CORS_ORIGINS` описать переменную в разделе «Переменные окружения» и в security-audit.

Критических уязвимостей, требующих немедленных правок, не выявлено. Рекомендации носят улучшающий характер и могут быть внедрены по приоритету (в т.ч. силами developer).

## Этап 8 (dev_plan_2): Загрузка .json на клиенте

**Дата проверки:** 2025-03-08  
**Цель:** Риски загрузки пользователем .json файла на клиенте — ограничение размера, парсинг без выполнения кода, отсутствие передачи JSON-файла на бекенд.

### Проверенные пункты

| Область | Результат | Детали |
|--------|-----------|--------|
| **constants.js** | ✅ | `MAX_JSON_FILE_SIZE = 2 * 1024 * 1024` (2 MB) задан и используется. |
| **slidesValidation.js** | ✅ | Используется только `JSON.parse(jsonString)`. Вызовов `eval`, `new Function`, `constructor` и иного выполнения кода не обнаружено. В коде есть явный комментарий: «Uses only JSON.parse() (no eval)». |
| **App.jsx — processJsonFile** | ✅ | Проверка `fileObj.size > MAX_JSON_FILE_SIZE` выполняется **до** вызова `FileReader.readAsText`. Расширение проверяется через `fileObj?.name?.toLowerCase().endsWith(".json")`. При успешном парсе — только `setSlides(result.data)` и при режиме byJson — `setJsonText(...)`; файл нигде не сохраняется и не отправляется на сервер. |
| **App.jsx — FormData** | ✅ | В `submit()` в FormData добавляются только `template` (файл PPTX) и `slides` (строка `JSON.stringify(payloadSlides)` из состояния приложения). Пользовательский .json файл в FormData не передаётся — на бекенд уходит только уже разобранное и пересобранное описание слайдов из клиентского состояния. |
| **Опасный парсинг во фронтенде** | ✅ | По всему `frontend/src` не найдено использований `eval(`, `new Function(`, `setTimeout(строка)`, `setInterval(строка)` или подобных конструкций с пользовательским вводом. |

### Дополнительные места использования пользовательского JSON

- **switchToBySlides** и **handleFormSubmit** (режим byJson): ввод из текстового поля `jsonText` обрабатывается только через `parseAndValidateSlides(jsonText)` → `JSON.parse` + валидация структуры и лимитов. Рисков не добавлено.
- **handleJsonFileInput / handleJsonDrop**: единственная обработка файла — вызов `processJsonFile(f)`; цепочка та же (проверка размера/расширения → readAsText → parseAndValidateSlides → setSlides/setJsonText).

### Вывод

Критических рисков не выявлено. Ограничение размера 2 MB применяется до чтения файла, парсинг выполняется только через `JSON.parse`, пользовательский .json файл на бекенд не передаётся и обрабатывается только на клиенте. Рекомендаций по обязательным правкам нет.
