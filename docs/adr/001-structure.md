# ADR 001: Структура проекта — монорепозиторий

**Статус:** Принято  
**Дата:** 2025-03-06  
**Контекст:** Этап 2.1 — выбор структуры проекта.

---

## Решение

Используется **единый репозиторий (монорепо)** с двумя корневыми приложениями: `frontend/` (React) и `backend/` (FastAPI). Документация в `docs/`.

---

## Контекст и обоснование

- **Требования:** раздельные фронтенд и бекенд, чёткий API-контракт между ними (PRD, план разработки).
- **Варианты:**
  1. **Монорепозиторий** — frontend и backend в одном репозитории, общая папка docs.
  2. **Раздельные репозитории** — отдельный репозиторий для frontend и для backend.

**Почему выбран монорепо:**

- Одна кодовая база упрощает синхронизацию версий и изменений API (изменение контракта видно в одном PR).
- Удобнее локальная разработка: клонирование одного репозитория, общие скрипты (например, docker-compose для frontend + backend).
- Для MVP команда/процесс предполагают совместную разработку фронта и бэка; независимый деплой двух репозиториев разными командами не обязателен.
- Документация (PRD, ADR, описание API) рядом с кодом обоих частей.

**Риски монорепо:** при росте команды и разделении зон ответственности возможен переход на раздельные репозитории; на текущем этапе это избыточно.

---

## Последствия

- CI/CD должен собирать и тестировать оба приложения (например, отдельные job’ы для frontend и backend).
- Структура папок и имена (`frontend/`, `backend/`, `docs/`) зафиксированы в архитектурном документе и README для онбординга.
