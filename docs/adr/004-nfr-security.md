# ADR 004: Нефункциональные требования и безопасность загрузки файлов

**Статус:** Принято  
**Дата:** 2025-03-06  
**Контекст:** Этап 2.3 — лимиты, таймауты, масштабируемость, безопасность.

---

## Решение

### Лимиты

- **Размер загружаемого файла (образца):** максимум **20 MB**. Обоснование: типовой корпоративный шаблон редко превышает 10–15 MB; 20 MB даёт запас без избыточной нагрузки на память и диск.
- **Количество слайдов в описании:** от **1 до 100** (MVP). Обоснование: PRD рекомендует порядок 100–200; для первого релиза 100 слайдов достаточно и обеспечивает предсказуемое время ответа.
- **Дополнительные лимиты** (см. api.md): длина заголовка слайда 500 символов, пункта — 2000 символов, не более 50 пунктов на слайд — для предотвращения переполнения и злоупотреблений.

### Таймауты

- **Таймаут запроса на генерацию:** **120 секунд**. При превышении сервер возвращает **504 Gateway Timeout** с понятным сообщением. Обоснование: генерация до 100 слайдов с применением стиля укладывается в 1–2 минуты на типовом железе; 120 с — разумный предел для веб-запроса.

### Масштабируемость

- **MVP:** один инстанс приложения; последовательная обработка запросов допустима. Очередь заданий и фоновые воркеры — пост-MVP при росте нагрузки.
- Рекомендация: не хранить загруженные файлы между запросами; после ответа удалять временные файлы (если они записывались на диск).

### Безопасность

- **Валидация загрузки файлов:**
  - Проверка **MIME-типа** и/или **сигнатуры** (OOXML — ZIP с определённой структурой); не полагаться только на расширение `.pptx`.
  - Жёсткий лимит размера (20 MB) на уровне приложения и при необходимости на уровне reverse proxy.
- **Обработка файла:** только как PPTX (парсинг через python-pptx). Не выполнять макросы, не извлекать и не выполнять произвольный код из загружаемых данных. Опасный контент внутри PPTX (например, OLE) не исполняется — библиотека работает с XML-представлением слайдов.
- **Валидация входных данных:** строгая проверка JSON описания слайдов (типы, границы массивов, длина строк) для предотвращения переполнения и некорректных данных.
- **Рекомендации при деплое:** HTTPS для production; при необходимости — лимит частоты запросов (rate limiting) в пост-MVP; секреты и конфигурация — через переменные окружения, не в коде.

---

## Последствия

- В конфигурации бекенда и (при необходимости) nginx/прокси зафиксировать лимит тела запроса 20 MB и таймаут 120 с.
- В коде — явная проверка типа и размера файла до парсинга; единая точка возврата ошибок с кодами 400/413/500/504 и сообщениями из api.md.
- Документация для разработчика и дизайнера: architecture.md, api.md и данный ADR.
