# ADR 002: API — один эндпоинт генерации, multipart/form-data

**Статус:** Принято  
**Дата:** 2025-03-06  
**Контекст:** Этап 2.2 — проектирование бекенда и API-контракта.

---

## Решение

- Один эндпоинт: **POST /api/presentation/generate**.
- Тело запроса: **multipart/form-data** с двумя полями:
  - `template` — файл образца (PPTX);
  - `slides` — JSON-строка с описанием слайдов (массив объектов `{ "title": string, "items": string[] }`).
- Успешный ответ: **200 OK**, тело — бинарный PPTX, заголовок `Content-Disposition: attachment; filename="presentation.pptx"`.
- Ошибка: **4xx/5xx**, тело — JSON с полем `detail` (строка или массив сообщений).

---

## Контекст и обоснование

**Альтернативы:**

1. **Два запроса:** сначала загрузка образца (POST /api/template), возврат id; затем POST /api/presentation/generate с template_id и JSON слайдов.  
   Минус: усложнение (хранение файла, сессия/токен), для MVP избыточно.

2. **Один запрос, multipart (выбрано):** файл и JSON в одном запросе.  
   Плюс: атомарность, нет состояния на сервере между запросами, проще фронтенд и бекенд.

3. **Base64 файла в JSON:** весь запрос в JSON, файл в base64.  
   Минус: увеличение размера запроса, лишняя нагрузка на парсинг; для бинарных файлов multipart предпочтительнее.

**Выбор multipart/form-data** соответствует типичной практике загрузки файлов в REST API и поддерживается FastAPI и браузерным FormData без дополнительных библиотек.

---

## Формат поля `slides`

```json
[
  { "title": "Заголовок слайда 1", "items": ["Пункт 1", "Пункт 2"] },
  { "title": "Слайд 2", "items": [] }
]
```

- Минимум 1 слайд, максимум — см. лимиты в api.md (100 для MVP).
- Пустые `title` и `items` допустимы по user stories.

## Последствия

- Фронтенд формирует FormData, добавляет файл и строку JSON; бекенд парсит multipart и валидирует оба поля.
- Документация контракта (коды ответов, ошибки, лимиты) сосредоточена в [api.md](../api.md).
